# NNUEネット構造の比較 — 水匠5(旧型) と 新型ネット

水匠5の評価関数（HalfKP 256x2-32-32）と、最新やねうら王が対応する新型ネット構造を
並べて比較した学習用資料です。構造定義は YaneuraOu の
`eval/nnue/architectures/*.h` から引用しています。

---

## 1. 共通の枠組み（HalfKP系）

どのHalfKP系ネットも、入力は同じ **HalfKP特徴量** です：

- 入力one-hot次元 = `SQ_NB(81) × fe_end(1548) = 125,388`
- 「自玉位置 × 玉以外の駒」の組み合わせ
- 同時に1になるのは最大38個（スパース）

違いは **①特徴量変換器の出力次元（TF dim）** と **②後段の密結合層の形** です。

```
HalfKP(125388, スパース)
   │
[FeatureTransformer] 125388 → TF_dim     ← ここが評価関数ファイルの大半を占める
   │ 2視点で TF_dim×2
   ▼
[ClippedReLU]→[Affine]→…→[Affine →1]      ← 後段の小さな密結合層
   ▼
評価値
```

---

## 2. 構造一覧（やねうら王 master 時点）

| 構造名 | TF dim | 1層目(入力→) | 2層目 | 出力 | 1層目入力の最適化 |
|---|---|---|---|---|---|
| **halfkp_256x2-32-32（水匠5）** | **256** | 512→32 | 32→32 | 32→1 | 密 (dense) |
| halfkp_512x2-16-32 | 512 | 1024→16 | 16→32 | 32→1 | SparseInput |
| halfkp_1024x2-8-32 | 1024 | 2048→8 | 8→32 | 32→1 | SparseInput |
| halfkp_1024x2-8-64 | 1024 | 2048→8 | 8→64 | 64→1 | SparseInput |

同系統の別系列（参考。互換性は無い）：
- `halfkpe9_256x2-32-32` … HalfKPに王手情報(E9)を加えた特徴量
- `halfkpvm_256x2-32-32` … HalfKP variant
- `kp_256x2-32-32` … KP特徴量（玉と駒、Half無し。評価関数1MB未満と軽量）
- `sfnnwop-1536` … Stockfish系の大型構成

---

## 3. パラメータ規模の比較（feature transformerが支配的）

FeatureTransformerの重み = `125,388 × TF_dim × 2byte(int16)`。
**評価関数ファイルのサイズはほぼこの値で決まります**（後段の密結合層は数十KB〜数百KBで誤差）。

| 構造 | TF dim | FT重み（概算） | 評価関数ファイル概算 |
|---|---|---|---|
| **halfkp_256x2-32-32（水匠5）** | 256 | 125388×256×2 ≈ **64.2 MB** | **約64 MB**（実測 64,217,066 byte） |
| halfkp_512x2-16-32 | 512 | 125388×512×2 ≈ 128.4 MB | 約128 MB |
| halfkp_1024x2-8-32 | 1024 | 125388×1024×2 ≈ 256.8 MB | 約257 MB |
| halfkp_1024x2-8-64 | 1024 | 同上 ≈ 256.8 MB | 約257 MB |

## 4. 進化の方向性（読み取れる設計思想）

水匠5(256x2-32-32)から新型への変化を見ると、明確な傾向があります：

1. **特徴量変換器を大型化**：256 → 512 → 1024
   - 巨大ルックアップテーブル側に「知識」を多く持たせる
   - 差分更新できる部分なので、大きくしても1手あたりの追加コストは小さい

2. **後段の密結合層は逆に小型化**：1層目 32 → 16 → 8
   - 全結合は差分更新できず毎回フル計算 → ここを小さくして高速化
   - `AffineTransformSparseInput`（疎入力アフィン変換）で更に高速化

→ つまり **「賢く大きい変換器 ＋ 軽い後段」** にシフト。
評価精度を上げつつ、1局面あたりの計算コスト増を抑える設計です。

### 水匠5(旧型・小型)の特徴
- ファイルが小さい(64MB)、後段が密(32-32)で計算は単純
- **1局面の評価が軽い → NPS（読みの速さ）が高い**
- 評価精度では新型に劣るが、**同じ時間で深く読める**のが武器
- 高クロック・多スレッド・大L3キャッシュ機では、この「速さ」が最も映える

---

## 5. 互換性の注意（再掲）

上記の各構造は **構造ハッシュ(kHashValue)が全て異なる**ため、相互に互換性はありません。

- 水匠5の `nn.bin` は **halfkp_256x2-32-32 のエンジンビルドでのみ**読み込める
- 512x2 / 1024x2 / halfkpe9 / kp256 等のビルドに入れても、ハッシュ不一致で起動失敗する

評価関数ファイルとエンジンビルドの**構造を必ず一致**させてください。
